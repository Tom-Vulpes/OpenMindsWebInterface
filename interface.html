The key principle

You cannot change the person. You cannot change the condition. But you can design the environment differently.

Your hypothesis

Phase 2

Investigation

Before you design anything, test your hypothesis against reality. Is this challenge real? Who experiences it? What has already been tried? Intellectual honesty lives here.

Your hypothesis will appear here once you've written it in Phase 1.

Is this actually true?

What evidence can you find that this challenge is real? What do people who live with it say?

Who experiences this?

How many people? In what contexts? Are there differences between groups?

What has already been tried?

What existing solutions, technologies, or approaches exist? What worked? What didn't — and why?

Does your hypothesis hold up?

After investigating, what's your verdict?

Why this phase matters

The most common design failure is solving the wrong problem brilliantly. Investigation is what separates a genuine response from a clever gadget nobody needs.

Research strategies

  • Talk to people who are directly affected
  • Search for existing research and reports
  • Look for organisations working on this
  • Ask who benefits from this staying unsolved
  • Check what's failed before, and why

On using AI here

AI can help find sources and explore existing knowledge — but evaluate what it returns. Verify. Follow citations. Prioritise talking to actual people over AI summaries.

Phase 3

What tools make possible

Now that you understand the challenge, think about what a response could look like. Technology is one option — sometimes the right prototype has no technology at all. If you do use tech, explore what it actually makes possible before you commit to a direction.

💡
Technology is a tool, not the answer

The best response to a challenge might be a policy change, a piece of signage, a different layout, a trained person, or a simple mechanical solution. A working prototype built with the Open Minds board is a fast, responsive way to test whether a technology-based approach adds real value — not a commitment to tech for its own sake.

Phase 4

Design & Prototyping

Build a working prototype of your response — something you can put in someone's hands. The goal isn't a finished product. It's something responsive enough to test with real people and learn from.

1

Connect your board

Plug your Open Minds board into your computer with a USB cable. It should appear as a removable drive called CIRCUITPY — like a USB memory stick.

Not seeing CIRCUITPY? Try a different USB cable — some cables are charge-only and don't carry data. If the board shows up as a different drive name, that's fine too.

Then click ⬡ Connect board in the editor above. A browser pop-up will ask you to select a serial port — choose the one that appears (usually named something like "USB Serial" or "Pico").

2

How the board runs your code

The board looks for a file called code.py on the CIRCUITPY drive and runs it automatically every time it starts up. When you click ▶ Run code, your code is saved to that file and the board restarts with your new program.

You don't need to understand everything — just know that what you write in the editor is exactly what the board will run.
3

Reading CircuitPython code

CircuitPython is designed to be readable. Here are the things you'll see most often:

# Lines starting with # are comments
# The board ignores them — they're notes for you
 
import time # Load a library (like an add-on)
 
while True: # Do this forever
    print("Hello!") # Show a message in the output
    time.sleep(1) # Wait 1 second

Indentation matters. The four spaces before print and time.sleep tell CircuitPython those lines are inside the while loop. If the spacing is wrong, the code won't work.

4

Try changing something — right now

Click Load this code below, then press ▶ Run code to see what happens in the output panel. The board isn't needed — this runs in your browser.

Example 1 — Change the message
import time
 
while True:
    print("Hello from my board!")
    time.sleep(1)

Now try changing "Hello from my board!" to your own message and run it again.

Example 2 — Change how fast it runs
import time
 
count = 0
 
while True:
    count = count + 1
    print("Count:", count)
    time.sleep(0.5)

Try changing 0.5 to 0.1 to make it count faster, or 2 to make it slower.

5

Touch pads making sounds — try it in your browser

Touch the pads below to hear different tones. This is what the touch pad code does on the board — except the board would produce a physical sound through the Audio Hat rather than through your computer speakers.

Each pad plays a different note. On the board, you'd use the Audio Hat to play tones through a speaker — the logic is the same.

The code that makes this work on the board
import board, time, touchio, pwmio
 
# Set up the speaker on GP16
speaker = pwmio.PWMOut(board.GP16, frequency=440)
 
# Each pad plays a different note frequency
notes = [262, 294, 330, 349, 392, 440, 494, 523]
pads = []
for pin in [board.GP2, board.GP3, board.GP4, board.GP5,
              board.GP18, board.GP19, board.GP20, board.GP21]:
    try:
        pads.append(touchio.TouchIn(pin))
    except:
        pads.append(None)
 
while True:
    for i, pad in enumerate(pads):
        if pad and pad.value:
            speaker.frequency = notes[i]
            speaker.duty_cycle = 32768 # 50% — makes sound
    time.sleep(0.01)
